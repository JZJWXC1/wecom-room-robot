# 项目 Agent 工作说明

本项目是「寓你住一起」长租公寓的企业微信/微信客服自动回复机器人，主要负责普通微信客户的房源咨询、房源表图片、房间视频、看房预约、合同联系方式和免押说明等回复。

当前默认服务对象为合作中介，运行配置视为 `audience=broker` / `broker profile`。后续可以保留租客侧 profile 设计，但未明确切换前，不得把租客口径、租客流程或租客私域规则混入当前中介客服链路。

## 基本原则

- 所有沟通、说明、提交总结都使用中文。
- 修改前先运行 `git status --short`，确认当前工作区状态。
- 不要回滚、覆盖或删除用户已有改动，除非用户明确要求。
- 不输出任何密钥、密码、token、access_token、App Secret、服务器登录信息。
- Python 读取明确为 UTF-8 的文本文件时，应显式指定 `encoding='utf-8'`。对于未知或历史编码文本，应先识别并保留原编码，不得为了读取方便批量改写文件编码或 EOL。二进制文件以及 Excel、图片、音视频等不得以 UTF-8 文本方式读取；CSV 应依据实际编码处理。PowerShell、`cat` 或终端出现中文乱码时，不得据此判断源文件损坏，应使用匹配编码重新读取验证。
- 能直接修复就直接修复，不只给计划；修完要测试、部署并验证。
- 若确实存在无法判断的关键信息，只问一个精准问题。

## 项目结构

- `app/main.py`：FastAPI 入口、微信客服回调、核心回复流程、素材直发逻辑。
- `app/services/llm.py`：大模型提示词和文本回复生成。
- `app/services/media_store.py`：房间图片、视频素材检索。
- `app/services/video_transcoder.py`：企业微信视频发送前的压缩转码。
- `app/services/feishu.py`：飞书表格和云盘同步。
- `app/services/inventory_image_sync.py`：飞书电子表格导出并渲染为房源表 PNG。
- `tests/`：单元测试。
- `room_database/`：服务器上的房源表、图片、视频素材目录。

## RAG 链路约束

- 客服可见回复必须走 Agentic RAG 链路：问题重写/意图分析 -> Planner/工具 -> 最终自检 -> 发送。
- 不允许新增绕过 RAG 的客服回复分支；如果确实需要修改非 RAG 链路，必须先说明文件属于哪段链路以及原因。
- 每次改代码时，说明改动归属：问题重写/意图分析、Planner、结构化会话记忆、自检回流、发送阶段、测试覆盖、房源/素材同步、运维部署。
- 旧固定规则只能作为最后安全阀，不能代替 RAG 链路级修复。
- 重构或新增 RAG 链路能力时，必须同步删除、停用或迁移会影响判断的旧回复规则、旧兜底分支和旧意图分支；不能让旧规则与新 Orchestrator 同时抢判断。若某个旧规则暂时不能删除，必须说明它保留在哪个 RAG 阶段、为什么保留、由哪些测试覆盖。
- 房源、价格、房态、密码、图片、视频事实只能来自最新房源表和素材库工具结果。

## 无人值守运行约束

- 任何 SSH、连接阿里云、上传服务器、重启服务、修改 systemd、飞书线上写入、飞书云盘写入、企业微信配置变更、线上数据修复或生产定时器修改，都必须先获得用户明确写出的 `APPROVE_DEPLOY`。没有 `APPROVE_DEPLOY` 时，只能做本地代码、文档、fixture、离线测试和只读分析。
- 服务器运行必须使用服务端长期凭证和 systemd，不依赖浏览器弹窗、扫码或人工确认。
- 飞书、企业微信、LLM、服务器 SSH 凭证必须通过环境变量、`.env` 或服务器密钥文件提供，不在代码和日志中输出明文。
- 同步任务、缓存刷新、机器人服务都要能通过 `scripts/server-ops.ps1` 非交互检查。
- 修改部署、同步、运维脚本时，优先保证幂等、可重复执行、失败可见。

## 修改流程

1. 先运行：

```powershell
git status --short
```

2. 阅读相关代码和测试，优先使用 CodeGraph；如果没有索引，再使用 `rg` 搜索。
3. 执行修改前，先审计本次改动会影响哪些模块、RAG 阶段、测试和线上行为；如果会影响其他模块，必须在改动说明和测试范围里体现。
4. 能并行开发和审计的任务要并行推进，但并入同一个工作树前，每个并行切片都要单独做一次审计，确认没有引入旧规则抢判断、跨模块副作用或测试缺口。
5. 使用 `apply_patch` 修改文件。
6. 修改后运行对应测试；影响核心回复逻辑时运行全量测试：

```powershell
$env:PYTHONPATH="$env:TEMP\wecom-room-robot-local-test-deps"
python -m pytest -q
```

7. 需要上线时，同步到阿里云 `/opt/wecom-room-robot`，在服务器运行测试，重启 `wecom-room-robot`，再做健康检查：

```bash
cd /opt/wecom-room-robot
PYTHONPATH=/tmp/wecom-room-robot-test-deps python3 -m pytest -q
systemctl restart wecom-room-robot
curl -sS http://127.0.0.1:8000/health
```

## 回复逻辑要求

- 问候语如“你好”“您好”“在吗”要拟人化回复，并引导客户直接问小区、房号、价格、空房、视频或房源表。
- 客户要“房源表”“表发一下”“发一下表”时，优先发送飞书电子表格渲染出来的 PNG 房源表。
- 客户说“笔记”时，不要回复“笔记”；本项目没有笔记，只有视频和房间详细信息。应理解为要视频或房间详细信息。
- 发送房间视频前，先发一句说明：`这是某某房间的视频。`
- 客户问两套、这几套视频时，要读取最近上下文，按最近提到的房源逐套匹配视频。
- 视频上传企业微信失败或文件过大时，优先在服务器用 ffmpeg 转码压缩，再重试原生视频发送；链接只是兜底。
- 还没空出的房子，如果客户问能不能看房、帮忙联系、预约看房，要引导联系：

```text
18758141785
13282125992
19941091943
```

- 合同、订房、交定金相关问题，也引导联系上述三个号码。
- 免押相关回复必须准确：
- 免押是支付宝芝麻信用“无忧住”服务，不是免费。
- 符合风控后，需支付押金金额 5.5%-8% 的免押服务费。
- 客户问“能不能免押”时，只发条件和自查方式。
- 客户问“免押是什么”或“服务费多少”时，再展开相应说明。
- 已关闭满意度追问，不要重新加回对话过程中的满意度追问。

## 飞书和素材要求

- 房源表以飞书电子表格为准，渲染 PNG 时要尽量保持和飞书表格一致，只截取有内容区域。
- 房源图片、视频从飞书云盘同步到服务器后，机器人运行时应直接使用服务器本地素材，不依赖本机。
- 房源表更新后，应检测飞书表格变动并重新生成最新 PNG。
- 不要把房源表格式改成纯文本列表，客户要表时优先发 PNG。

## 测试要求

- 修改回复意图、话术清洗、上下文读取、视频/图片发送逻辑时，至少更新或新增 `tests/test_wecom_kf.py`。
- 修改素材匹配时，至少更新或新增 `tests/test_media_store.py`。
- 修改飞书同步时，至少更新或新增 `tests/test_feishu.py` 或相关同步测试。
- 重要改动必须补测试或运行相关回归测试。
- 涉及中文输入的测试必须使用 UTF-8，不能出现乱码。
- 部署前本地全量测试要通过；部署后服务器全量测试也要通过。
- Agentic RAG 连续对话固定覆盖集通过后，还必须随机重新生成一组不同的 10 个问题做最终保底测试；随机 10 问一旦出现错误就停止测试，先审计失败归属并修复链路，不能继续把后续测试跑完来稀释失败。
- 服务器部署后至少检查服务状态、健康接口、定时器状态和无人值守凭证完整性。

## 输出要求

- 最终回复先说明修改原因和影响。
- 只展示关键修改片段，不粘贴整个文件。
- 明确说明测试结果、部署状态和健康检查结果。
- 如果有未完成或无法验证的地方，必须直接说明。
