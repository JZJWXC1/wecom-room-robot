# Milestone Map

本文件锁定 M0.7 之后的上线改造边界。当前文档只定义里程碑范围，不代表进入 M1，也不改变现有客服回复链路。

## M1: InventorySnapshot

目标：建立可审计、可原子切换的结构化房源快照。

范围：

- 飞书结构化房源同步。
- 字段校验：区域、小区、房号、户型描述、户型分类、押一付一、押二付一、看房方式密码、备注。
- `snapshot_id`、同步版本、生成时间和签名。
- 原子切换：新快照完整生成并校验通过后再替换当前快照。
- 同步报告：新增、更新、删除、字段异常、素材关联异常。
- 健康状态：最近成功同步时间、当前快照 ID、失败原因、可读性检查。

M1 明确不做：

- 不修改 Orchestrator、Planner、自检、发送链路。
- 不拆分 `app/main.py`。
- 不调整客服可见回复话术。
- 不修改企业微信发送行为。
- 不改素材 manifest 结构。

注意：`app/services/kf_orchestrator_flow.py` 是 M0 前已有的未跟踪内容，不属于 M1 交付范围。

## M2: MediaManifest

目标：把房间图片、视频素材整理为可校验、可查询、可增量更新的素材 manifest。

范围包括素材文件索引、房源 key 绑定、文件类型和大小校验、缺失素材报告、按需查询接口。M2 不改变 Planner 语义，只提供更可靠的素材事实来源。

## M3: 强类型契约和兼容适配器

目标：为问题重写、Planner、工具证据、自检、发送动作建立强类型输入输出契约。

范围包括 typed packet、兼容适配器、边界校验和最小迁移测试。M3 可以增加适配层，但不能新增绕过 Agentic RAG 的客服回复分支。

## M4: 对话和候选状态

目标：收敛黑匣子、候选集、confirmed room、pending video 等会话状态的读写边界。

范围包括状态读写 API、轮次记录、候选引用、上下文继承和追问恢复。M4 不负责房源事实生成，事实仍来自 M1/M2 的工具证据。

## M5: 自检与幂等发送

目标：让最终自检覆盖文本和动作，并让发送阶段可重试、可去重、可审计。

范围包括 RetryPacket、动作包一致性、自检失败回流、发送幂等键、重复发送保护和失败记录。

## M6: QA、故障注入、Shadow Mode

目标：建立离线、可复现、可分类的 QA 与故障注入能力，并在上线前进入 Shadow Mode。

范围包括固定 QA、随机 QA、网络隔离、fixture 来源、失败分级、shadow 对比报告和不可上线门禁。

## M7: 发布和回滚

目标：形成可审批、可回滚、可观测的发布流程。

范围包括 `APPROVE_DEPLOY` 门禁、服务器测试、健康检查、定时器检查、回滚包、版本记录和发布后观察。
