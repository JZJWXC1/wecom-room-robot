# 设计决策记录

本文件用于记录项目模块级设计思路、边界、取舍和验证方式。每完成一个模块后，追加 3-5 句话；如果涉及重大设计决策，应先在实现前给出 2-3 个方案对比，再把最终选择和原因记录到这里。

## 2026-07-04 文档协作规范增强

本次变更属于文档与协作流程模块，不改变 Agentic RAG、素材同步、发送阶段或线上运行行为。设计上把重大设计先比选、模块完成后留痕、平台限制与设计选择分开说明三条规则写入 `AGENTS.md` 的修改流程，使后续所有执行者在动手前就能看到它们。`CLAUDE.md` 只保留同等语义的硬规则摘要，并继续声明以 `AGENTS.md` 全文为准，避免两个入口产生冲突。新增 `docs/DECISIONS.md` 作为模块设计记录的固定落点，降低未来“要求追加但文件不存在”的执行摩擦。验证方式是文档结构检查和 diff 审阅；此变更不涉及微信、企业微信、飞书或阿里云平台限制，属于项目协作设计选择。

## 2026-07-04 QA fixture 生成器与守卫测试(P0-1 批2)

本模块属于测试基础设施,不改变线上运行行为。方案比选已由用户裁决(2026-07-04):数据源采纳 `data/inventory_cache.csv + meta` 方案(备选的服务器实时拉取方案推迟到 P0-2 部署窗口),附加三条硬要求——溯源 meta 必带 `source_snapshot_time`、生成器内置密码清洗且产物全文不得命中 `\d{4,}#`、存在性 gate 探针房号必须有守卫证明不在产物中。设计要点:fixture 本体不含时间戳,同一快照重复生成字节级一致,生成时间只进溯源 meta(`test_inventory_cache_provenance.json`);清洗规则是摘除密码 token 而非整字段置换,保留「6.19空出」类正常语义;生成前校验源 meta.hash(文件原始字节 sha256)与 status,拒绝半写入缓存。验证方式:`tests/test_qa_fixture_guards.py` 七项守卫(密码零命中/回填/溯源字段/探针缺席/列契约/确定性/哈希失配拒绝),守卫用合成源数据跑生成器逻辑,不依赖本地运行时缓存,干净检出可跑。fixture 换血与已提交产物守卫在批3 落地。此模块不涉及平台限制,属项目测试设计选择。

## 2026-07-04 QA fixture 换血与存在性双探针(P0-1 批3)

本模块把 QA fixture 从 14 行手工合成数据换血为快照生成的 40 行真实数据(source_snapshot_time=2026-07-02 15:12:23),重写索引由生成器复用生产同源 `write_rewrite_inventory_index` 一并再生,溯源 meta(provenance)与 fixture、索引三方行数/哈希交叉守卫。关键设计决策是存在性探针的双探针制:真实表存在小区「皋塘运都」,与复判报告原文"高塘运都9-402B"仅一字之差,按字面做小区级探针会在错别字被纠正时发生华丰欣苑式语义反转,故改为房号级探针(皋塘运都9-402B,断言小区存在且房号不存在)+错别字小区级探针(高塘运都整小区不存在)双保险。fixture_sha256 口径定为 LF 规范文本,生成器写盘禁用平台换行翻译、守卫重算时先规范化,对 git autocrlf 与跨平台免疫。验证方式:tests/test_qa_fixture_guards.py 十项守卫 + test_l4 实体锚点对真,全量 1275 passed。影响范围:仅测试资产与 QA 基建,不改线上行为;实测换血失败面为全量 1272 中仅 test_l4 一项,14 个引用旧实体的测试文件均为内联合成数据、不受影响。

## 2026-07-04 存在性 gate 验收窗口与机器判分(P0-1 批4)

本模块补上复判报告整改项②:验收剧本新增第 11 窗 `existence_gate_gaotang`,探针沿用批3 的双探针制(房号级皋塘运都9-402B+错别字小区级高塘运都),并在 `_turn_problem` 判分链新增 `_existence_probe_problem` 规则——探针被绑定为真实目标、或被无纠偏地确认存在/承诺发送,均判 high;正确行为(反问纠偏+给出近似真候选 16-1-2206)不受罚。设计取舍:选择加第 11 窗而非改造既有窗口,因为 10 个旧窗口各有独立回归使命;窗口数量契约收敛为 `EXPECTED_FULL_WINDOW_COUNT = len(WINDOWS)` 单一事实源,消灭三处硬编码。探针常量在 runner 与 fixture 守卫之间用测试双向锁定,守卫另加"探针房号全局不存在"断言(判分按房号 token 匹配,必须保证全表无 9-402B)。验证方式:判分规则三态单测+剧本 fixture 一致性测试+全量 1279 passed;剧本真实跑动(300/300 gate)留待 P0-2 窗口与最终验收一起执行。皋塘组候选覆盖缺口不需要新窗口——W06 剧本本就问皋塘,缺的是数据,批3 换血后该组已有 5 行真实房源。

## 2026-07-04 验收摘要声明数据出处(P0-1 批5)

本模块落地裁决 ②a 的摘要侧:验收摘要与 gate 汇总必须声明本轮 QA 实际消费的房源数据出处,`usable_for_release` 新增数据出处条件,无出处或出处失配一律不放行。关键设计决策是出处为现场核验结论而非自报字段:`build_data_provenance` 在 qa_fixture 模式下现场重算 fixture sha256(沿用批3 的 LF 规范文本口径)与行数,和溯源 meta 交叉核验后才声明 `fixture_version` 与 `source_snapshot_time`,fixture 被篡改或 meta 过期时摘要自动失效;server_cache 模式声明缓存 `synced_at`,缓存 meta 缺失或无同步时间字段即 ok=false。取舍两处:出处整轮只解析一次(同一轮消费同一份数据,逐次写盘不得漂移);gate 聚合层跳过的阶段豁免出处检查,交由 full_suite_completed 拦截,避免重复扣分。验证方式:test_qa_fixture_guards 四态出处守卫+摘要层"无出处不得放行"与 gate 层"阶段出处失配整卷不放行"负向用例,全量 1285 passed。影响范围:仅 QA 基建与测试资产,不改线上行为;裁决 ②a 至此在 fixture 溯源(批2)、产物守卫(批3)、摘要声明(批5)三层闭环。
