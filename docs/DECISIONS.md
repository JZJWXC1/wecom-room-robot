# 设计决策记录

本文件用于记录项目模块级设计思路、边界、取舍和验证方式。每完成一个模块后，追加 3-5 句话；如果涉及重大设计决策，应先在实现前给出 2-3 个方案对比，再把最终选择和原因记录到这里。

## 2026-07-04 文档协作规范增强

本次变更属于文档与协作流程模块，不改变 Agentic RAG、素材同步、发送阶段或线上运行行为。设计上把重大设计先比选、模块完成后留痕、平台限制与设计选择分开说明三条规则写入 `AGENTS.md` 的修改流程，使后续所有执行者在动手前就能看到它们。`CLAUDE.md` 只保留同等语义的硬规则摘要，并继续声明以 `AGENTS.md` 全文为准，避免两个入口产生冲突。新增 `docs/DECISIONS.md` 作为模块设计记录的固定落点，降低未来“要求追加但文件不存在”的执行摩擦。验证方式是文档结构检查和 diff 审阅；此变更不涉及微信、企业微信、飞书或阿里云平台限制，属于项目协作设计选择。

## 2026-07-04 QA fixture 生成器与守卫测试(P0-1 批2)

本模块属于测试基础设施,不改变线上运行行为。方案比选已由用户裁决(2026-07-04):数据源采纳 `data/inventory_cache.csv + meta` 方案(备选的服务器实时拉取方案推迟到 P0-2 部署窗口),附加三条硬要求——溯源 meta 必带 `source_snapshot_time`、生成器内置密码清洗且产物全文不得命中 `\d{4,}#`、存在性 gate 探针房号必须有守卫证明不在产物中。设计要点:fixture 本体不含时间戳,同一快照重复生成字节级一致,生成时间只进溯源 meta(`test_inventory_cache_provenance.json`);清洗规则是摘除密码 token 而非整字段置换,保留「6.19空出」类正常语义;生成前校验源 meta.hash(文件原始字节 sha256)与 status,拒绝半写入缓存。验证方式:`tests/test_qa_fixture_guards.py` 七项守卫(密码零命中/回填/溯源字段/探针缺席/列契约/确定性/哈希失配拒绝),守卫用合成源数据跑生成器逻辑,不依赖本地运行时缓存,干净检出可跑。fixture 换血与已提交产物守卫在批3 落地。此模块不涉及平台限制,属项目测试设计选择。
