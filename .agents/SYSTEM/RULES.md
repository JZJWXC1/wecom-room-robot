# 项目规则

## Agentic RAG 改动边界

- 所有客服机器人相关代码改动，默认只允许发生在 Agentic RAG 链路内，包括：问题重写/意图分析、结构化会话记忆、Planner、工具调用、工具证据、Planner 工具后回复生成、自检回流、房源表/素材库事实读取、欢迎语入口。
- 不允许新增、恢复或保留直接绕过 RAG 链路生成客服回复的分支、模板短路、固定规则直出、旧逻辑前置拦截。
- 如果某个请求确实需要临时绕过 RAG 链路，必须在改代码前明确说明：
  - 绕过的是哪一段 RAG 链路；
  - 为什么不能走 RAG；
  - 会影响哪些客服场景；
  - 如何回收或重新接回 RAG 链路。
- 只有在用户明确同意后，才可以实施绕过 RAG 的代码改动。
- 房源、价格、房态、密码、图片、视频、房源表等事实信息，必须通过 RAG 链路里的工具和事实证据读取，不允许由绕过分支、LLM 猜测或固定模板直接编造。
- 所有兜底回复、确定性回复、硬规则生成回复、工具证据兜底回复，以及视频/图片/房源表等发送动作，都必须进入最终自检；自检要检查事实一致、动作一致、上下文连贯和拟人化口吻，通过后才能发送。

## RAG 改动说明要求

- 每次完成代码改动后，必须说明每个被改动文件属于 Agentic RAG 的哪一段链路。
- 文件归属说明要使用用户能看懂的链路名称，例如：企业微信入口、上下文读取、问题重写/意图分析、结构化会话记忆、Planner、工具调用、房源表/素材库事实读取、Planner 工具后回复生成、自检回流、发送阶段、欢迎语入口、测试覆盖。
- 如果某个文件同时跨多个链路，必须列出所有相关链路，并说明这次改动主要影响哪一段。
- 如果文件不属于 RAG 链路但确实需要改动，必须在改动前说明原因并取得用户同意。

## Agentic RAG 模块化目标提示词

后续整理和重构客服机器人时，必须把下面内容作为目标提示词执行：

> 当前目标是把「寓你住一起」企业微信客服机器人整理成清晰、可查询、可测试的 Agentic RAG 链路工程。拆文件和模块化不但不冲突，反而更符合稳定可用目标。当前 `app/main.py` 承担了过多职责：问题重写、意图分析、Planner、工具执行、Planner 工具后回复生成、自检、发送、上下文保存混在一起，后续排查“到底是哪条链路错了”会越来越困难。
>
> 重构时必须先用 CodeGraph 查看调用关系和影响范围，再按 Agentic RAG 链路拆分模块。拆分必须按链路职责进行，不能按零散功能随意切碎。推荐目标文件结构如下：
>
> - `kf_rewrite_flow.py`：问题重写、意图分析、实体归一、约束证明。
> - `kf_planner_flow.py`：Planner 输入输出、动作规划、工具证据后的 `reply_text` 生成、RetryPacket。
> - `kf_tool_flow.py`：房源表、素材库、规则库工具执行和证据汇总。
> - `kf_selfcheck_flow.py`：最终自检、拟人化、上下文连贯、自检回流。
> - `kf_send_flow.py`：发送文本、图片、视频、房源表动作。
> - `kf_turn_flow.py`：一轮企业微信消息从输入到发送的总编排。
> - `kf_context_memory.py`：继续保留结构化会话黑匣子。
>
> 关键约束：拆文件可以，但不能改变 RAG 链路顺序，不能新增绕过 RAG 的客服回复分支，不能让确定性规则直接跳过最终自检。Planner 的正确顺序是：读取结构化任务包，先规划并使用房源表、素材库、规则库等工具查询或获取事实证据，再根据工具调用结果生成客户可见 `reply_text`；不得写成“Planner 先生成 `reply_text` 再执行工具”。拆分后的每个模块都必须有清晰输入、输出和测试覆盖；CodeGraph 查询时应该能快速定位“问题重写、Planner 工具规划、工具证据、Planner 工具后回复生成、自检、发送、黑匣子”分别在哪里发生。
