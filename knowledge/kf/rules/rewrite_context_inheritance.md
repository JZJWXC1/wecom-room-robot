# 上下文继承

连续对话里的短句通常是在补充上一轮需求。问题重写/意图分析必须读取黑匣子原始对话、上一轮重写结果、候选列表、已确认房源、最近发送素材和 pending 视频状态，再判断当前输入是新问题、追问、补充、继续发送素材，还是换房源/换区域。

口语指代必须结构化：
- “上一个、上个、上一套、就这个、就这套、这套、那套、刚才那个”优先绑定最近 confirmed room 或最近发送过素材的房源。
- “第一套、1和5、前两套、这三套、都发”必须绑定上一轮候选编号，不能重新全表搜索。
- “星桥那个、棠闰府那套”这类“小区提示 + 那个/那套”要先做实体归一，再只在上一轮候选里唯一匹配；候选里不唯一或没有命中时必须追问。

常见错别字必须先归一再绑定：
- 棠闰府、堂润府、棠润肤 -> 棠润府。
- 用户下文说“棠闰府那套视频发我”，如果上文机器人发过棠润府候选且唯一，就绑定这套；如果上文有多套棠润府，追问具体房号。

区域别名必须来自最新房源索引：
- 万达默认归一到“拱墅万达 / 北部软件园 / 城北万象城”相关区域，不追问城市。
- 新天地、东新园、杭氧等必须按房源索引里的标准区域归一。
- 石桥既可能是区域，也可能指石桥铭苑小区；如果上下文没有明确是区域还是小区，必须基于真实候选追问。

如果用户说“原视频、高清视频、不要压缩、太糊了、能保存转发的”，这是素材交付方式升级。问题重写应输出 `wants_original_video=true`，并保留已绑定房源或候选编号；后续由 Tool Resolver 查原视频、下载链接或素材页证据。
