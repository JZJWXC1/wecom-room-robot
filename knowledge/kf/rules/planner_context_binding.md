# Planner 上下文目标绑定

Planner 只接收问题重写层产出的结构化任务包，不重新解释用户意图。
凡是“这套、那套、上一个、上个、就这个、就这套、刚才那个、星桥那个、棠闰府那套”这类口语指代，Planner 必须使用任务包里的 `target_rows`、候选编号、标准小区名和标准房号执行；如果目标不唯一，只能返回内部缺失证据给问题重写层，不能自己猜房源，也不能直接乱发素材。

用户说“1和5视频、前两套视频、这三套视频、都发视频”时，Planner 必须按上一轮候选编号调用素材工具；不能把编号当房号，也不能重新按整张房源表泛搜。

如果当前任务已经带有新的区域、预算、户型、小区范围，且用户问的是“附近有没有、这边有哪些、预算多少以内、某区域几室”等新范围查询，即使问题重写层标了 `context_reference=true`，Planner 也不能把旧 `confirmed_room` 混入本轮 `target_rows`。本轮查询必须以最新房源搜索结果为准。

工具结果已经证明视频、图片或房源表可发送时，Planner 生成的客户可见回复必须保留这些动作，并用自然话术说明：
- 视频：这是某某小区+房号的视频。
- 图片：这是某某小区+房号的图片。
- 房源表：房源表发你了，你可以让客户先整体看一下。

如果最终自检只因为“话术模板感、口吻不够自然、上下文接话不顺”失败，Planner 重规划时要改写话术，但不能丢掉已经由工具证据证明可发送的图片、视频或房源表动作。只有事实不一致、目标不明确、文件不存在、动作和文字矛盾时，才允许拦截动作。

原视频/高清视频/不要压缩/客户要保存转发，属于素材需求升级。Planner 必须优先找原始视频交付方式；如果只能发压缩视频，要明确说明，并给出可保存转发的替代方案，例如原视频下载链接或按文件发送。
